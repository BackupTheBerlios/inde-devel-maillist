> is there any reason for not subclassing Settings for ProjectSettings?

Actually there is a reason.
Regarding to Settings r24, subclassing from Settings had been ok, but i
do not see, why i should overload ProjectSettings with color settings
and so on. Also the Format settings will be deleted.

If we had a common base for the settings, e.g. CommonSettings.h, which
would implement reading/writing the StringValues and IntValues, give
functionality for parsing and writing, this would be a very fine super
class.

But Settings, as you are writing it, will hold many special
functionality, which isn't interesting for the project settings, and i
will also implement some functions, which aren't important for Settings.

Giving Settings and ProjectSettings a common base class, would be clear
so show that they relate and do basically similar things. But from the
code view, there is no reason to give those classes more common
functionality than FXSettings. I really can't imagine the following
scenario for any job:

snip >>>>>
CommonSettings	*settings;

Settings	*indeSettings = new Settings(...);
ProjectSettings *projectSettings = new ProjectSettings(...);

if (condition)
	settings = indeSettings;
else
	settings = projectSettings;

settings->doJob();
<<<<< snap

	Jochen
